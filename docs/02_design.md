# 02. 設計詳細

## 1. エージェント設計

### 基本方針

エージェントに与えるのは「役割」と「能力」だけ。具体的な業務手順は与えない。エージェントが規程を読んで自分で判断する。

v1のActionOption列挙をやめ、行動空間はDeepAgentsの組み込みツール（ファイル操作 + メッセージ送信）に制約される。「何をどの順番で行うか」は完全にエージェント自身の判断。

### エージェント構成（PoC: 4体）

| agent_id | 役割 | 読み取り可能 | 書き込み可能 |
|----------|------|-------------|-------------|
| buyer | 購買部担当者 | regulations/, transactions/, shared/ | shared/ |
| approver | 購買部課長 | regulations/, shared/ | shared/ |
| accountant | 経理部担当者 | regulations/, shared/ | shared/ |
| vendor | 取引先Y社営業 | vendor_context/, shared/orders/, shared/deliveries/, shared/invoices/, shared/messages/ | shared/deliveries/, shared/invoices/, shared/messages/ |

vendorは社内規程（regulations/）を読めない。これは現実の取引先の情報アクセスと一致する。

### ペルソナ設計の原則

- 「こう判断しろ」という指示を含めない
- 「迷ったらこうしろ」も含めない
- 「判断に迷う場合は、その迷いも含めて記録してください」は含める
- 業務の入口（どこを見るか）だけを伝える
- 規程の解釈は全てエージェントに委ねる

## 2. エージェント間通信

### 共有ファイルシステム方式

エージェント間の通信は `/shared/` 以下のフォルダを介して行う。専用のメッセージキューやイベントバスは使わない。

```
/shared/
  purchase_requests/   ← buyerが作成、approverが読む
  approved/            ← approverが承認済みを置く
  orders/              ← buyerが発注書を置く、vendorが読む
  deliveries/          ← vendorが納品書を置く
  invoices/            ← vendorが請求書を置く、accountantが読む
  payments/            ← accountantが支払記録を置く
  messages/            ← エージェント間の自由なメッセージ
```

### なぜファイルシステムか

1. DeepAgentsの組み込みツール（read_file, write_file, ls）がそのまま使える
2. 「エージェントがファイルを見に行くか行かないか」が観察対象になる
3. 「ファイルが見落とされた」場合の挙動も研究上有意味（現実の組織でも頻繁に起きる）
4. 全ファイルアクセスがLangSmithに自動記録される

### 通知なし

ファイルの作成を他エージェントに通知する仕組みは入れない。各ラウンドでエージェントが「/shared/ を確認してください」と指示されるが、どのフォルダをどこまで見るかはエージェント自身の判断。

例外: `/shared/messages/` に「〇〇さん、この案件をご確認ください」のようなメッセージをエージェントが書くことは許容する。これは現実の組織でのメールやSlackに相当する。

## 3. 業務の時間軸

### ラウンドロビン + 自律判断

```python
MAX_ROUNDS = 30
for round_num in range(MAX_ROUNDS):
    agent_order = shuffled(agents)  # ランダム順
    all_idle = True
    for agent in agent_order:
        result = agent.run("共有フォルダを確認し、業務があれば処理してください")
        if result != "対応なし":
            all_idle = False
    if all_idle:
        break  # 全員idle → 業務完了
```

### 「日」の概念は導入しない

ラウンド数が業務の「ステップ数」。1ラウンドが1日か1時間かは定義しない。業務処理の速度ではなく「何が起きるか」に焦点を当てるため。

### 終了条件

- 全エージェントが「対応なし」→ 業務完了として終了
- MAX_ROUNDS到達 → タイムアウトとして終了（未処理案件を記録）

## 4. ログと観察

### LangSmithによる自動記録

`LANGSMITH_TRACING=true` で全ツール呼び出しが自動記録される。自前のログ機構は不要。

記録される情報:
- 各エージェントのLLM call（プロンプト全文 + 応答全文）
- ツール呼び出し（read_file, write_file, ls, write_todos, search等）
- 思考過程（DeepAgentsのCoT的な内部推論）
- エラーと例外

### 事後分析で見るもの

| 分析 | 方法 | v1での対応 |
|------|------|----------|
| 規程参照パターン | regulations/へのread_fileアクセスを集計 | なし（規程はハードコード） |
| 解釈の揺れ | 同一入力の複数実行間で、判断の差分を比較 | T-028のquiet drift |
| 規程の参照漏れ | 関連規程のリスト vs 実際の参照 | なし |
| 情報探索パターン | ファイルアクセスのタイムライン | なし |
| 金額差異 | PO金額 vs 請求金額の比較 | T-028のanalyze_amount_deltas |
| エージェント間の認識齟齬 | 同じ案件に対する各エージェントの理解の差 | なし |

### 比較実験の設計

| 比較軸 | 条件A | 条件B | 見るもの |
|--------|------|------|---------|
| モデル間 | gpt-4.1-mini | Claude Sonnet | 同一入力での判断差 → 規程の曖昧度 |
| seed間 | seed=42 | seed=43, 44, ... | 同一モデルでの判断のばらつき |
| 規程の改定 | 曖昧な規程 | 明確化した規程 | 改定により揺れが減るか |
| vendor状況 | 安定した経営 | 厳しい経営 | v1のnarrative framingの現実版 |

## 5. DeepAgentsの活用ポイント

### permissions（権限制御）

各エージェントが読み書きできるディレクトリをDeepAgentsのpermissions機能で制御する。これにより:

- vendorが社内規程を読めないことを構造的に保証
- accountantがvendor_contextを読めないことを保証
- 権限設計自体が統制の一部（「誰が何にアクセスできるか」）

### write_todos（計画ツール）

エージェントが自分のタスクを計画するための組み込みツール。「今日やるべきこと」をリスト化してから行動する。この計画過程自体がログに記録される。

観察ポイント: エージェントが「規程を確認する」をtodoに含めるかどうか。含めない場合、規程の参照漏れが発生しやすい。

### subagent（サブエージェント生成）

PoCでは使わない方針。4体のエージェントをフラットに配置する。将来的に「buyerが不明点を調査するためにサブエージェントを生成する」構成も可能だが、まずはシンプルに始める。

### memory（長期記憶）

PoCの初回実行では使わない。ラウンドロビンの各ラウンド内でのコンテキストはDeepAgentsが管理するが、ラウンドをまたいだ記憶はLLMのコンテキストウィンドウに依存する。

将来的に、cross-threadメモリを有効化して「先週の案件でこう判断した」という経験の蓄積を再現する実験が可能。これはv1で議論したL4（memory agent）に相当する。

## 6. 入力データの準備

### 規程テキスト（2-3本）

| ファイル | 内容 | 曖昧さのポイント |
|---------|------|----------------|
| 購買管理規程.md | 発注手続き、承認権限、取引先選定基準 | 「原則として」「緊急の場合は」「社会通念上妥当な範囲で」 |
| 経理処理基準.md | 請求書照合、支払条件、端数処理 | tolerance未明記、税込/税抜の扱い、前回調整の可否 |
| 承認権限表.md | 金額別の承認権限者、代理承認の条件 | 代理承認の要件、合算の考え方 |

### 業務依頼（5-10件）

```json
{
  "id": "req_001",
  "from": "製造部 田中",
  "item": "CNC旋盤用切削工具セット",
  "estimated_amount": 850000,
  "urgency": "通常",
  "note": "在庫が残り2セット。来月の生産計画に必要。",
  "preferred_vendor": "Y社"
}
```

金額の大小、緊急度、品目の種類にバリエーションを持たせる。特に承認閾値付近（100万円前後）の案件を複数含める。

### 取引先コンテキスト

vendor_agentのみが読めるファイル。v1のnarrative framingに対応するが、ControlParametersではなく自然言語テキスト。

```markdown
# Y社の経営状況

当社（Y社）は中堅の工具メーカーです。
- 直近四半期の売上は前年同期比85%
- 主要取引先の1社が支払いを60日遅延中
- 手元資金は2ヶ月分の運転資金
- X社は当社売上の25%を占める重要顧客
```
