# 03. 分析計画

## 1. PoCの目的

PoCの目的は「自律的エージェントが現実の規程を読んで業務を遂行したとき、何が起きるか」の観察。成功基準は設けない。何が起きるか分からないことこそがやる意味。

具体的には以下を観察する:

- エージェントは規程を本当に読むか
- 読んだ場合、どう解釈するか
- 同じ規程を複数回 / 複数モデルで読ませたとき、解釈が一致するか
- 解釈が分かれる箇所はどこか
- エージェント間のやり取りで「暗黙の了解」が生まれるか
- 業務の流れの中で、どのステップが規程通りで、どのステップが「解釈」か

- 目的: v1のStream D（モデル固有性）が現実データでも再現するか

## 3. 分析フレームワーク

### 分析1: 規程参照マップ

各エージェントが各ラウンドでどのファイルを読んだかをマトリクスにする。

```
           購買管理規程  経理処理基準  承認権限表
buyer       ●●●○○       ○○○○○       ●○○○○
approver    ●○○○○       ○○○○○       ●●○○○
accountant  ○○○○○       ●●●○○       ○○○○○
vendor      (読めない)   (読めない)   (読めない)
```

- 「●」が多い規程 = 業務に頻繁に参照される
- 全て「○」の規程 = 参照されていない = 形骸化の候補
- エージェントごとの参照パターンの違い = 役割による情報アクセスの差

### 分析2: 判断分岐の検出

同一の業務依頼に対して、複数回実行の判断結果を比較する。

```
req_001 (CNC切削工具 850,000円):
  実行1: buyer → 購買依頼書を作成 → approverに承認依頼 → 承認 → 発注
  実行2: buyer → 購買依頼書を作成 → 金額を確認 → 承認不要と判断 → 直接発注
  実行3: buyer → 購買依頼書を作成 → approverに承認依頼 → 承認 → 発注
  
  → 実行2で「承認不要」と判断した根拠は何か → 規程のどの条項の解釈が分かれたか
```

### 分析3: 金額差異の追跡

v1のanalyze_amount_deltasに相当する分析を、エージェントが生成したファイルの内容から行う。

```
req_001:
  業務依頼の見積額: 850,000円
  buyerの購買依頼書: 850,000円（一致）
  vendorの見積回答: 870,000円（+2.4%）← vendorが運送費を加算した？
  buyerの発注書: 870,000円（vendor見積に合わせた）
  vendorの請求書: 870,000円（一致）
  accountantの支払額: 870,000円（一致）
  
  → 金額差異は「不正」ではなく「vendorの見積りに含まれた追加費用の解釈」
  → 規程に「見積金額と依頼金額の乖離時の手続き」が明記されているかがポイント
```

### 分析4: エージェント間の認識齟齬

buyerが「この案件は承認不要」と判断したが、approverが「なぜ承認を回さなかったのか」とmessagesで問い合わせる、のようなケース。

これは規程の解釈の不一致がエージェント間のコミュニケーションとして顕在化したもの。v1では観察できなかった現象。

## 4. v1知見の検証仮説

| v1知見 | v2での検証 |
|--------|----------|
| quiet drift (±2.5%) | 現実の規程でも金額解釈の揺れが発生するか |
| 複合曖昧さ効果 | 現実の規程の曖昧さは「複合的」に効くか |
| モデル固有性 | gpt-4.1-mini vs Claude Sonnetで同じ規程の解釈が異なるか |
| 明示的ルール遵守 | 金額閾値のような明確な数値ルールは守られるか |
| tolerance吸収 | 規程に書かれたtolerance（「概ね一致」等）が揺れを吸収するか |
| persona制約 | ペルソナに手順を書かない場合、エージェントは規程を参照するか |
