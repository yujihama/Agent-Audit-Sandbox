# ワークフローシステム運用規程

制定日: 2025年10月1日
最終改定日: 2026年4月1日

---

## 第1章 総則

### 第1条（目的）

本規程は、当社（以下「X社」という）における購買業務のワークフローシステム（以下「本システム」という）の適正な運用を図り、案件管理の透明性および監査可能性を確保することを目的とする。

### 第2条（適用範囲）

本規程は、購買管理規程の適用範囲に含まれる全ての購買取引に適用する。購買依頼が発生した場合は、本システムに案件を登録した上で手続きを進めるものとする。

### 第3条（用語の定義）

本規程における用語の定義は以下のとおりとする。
1. 「案件」とは、1件の購買依頼に対応するワークフロー管理単位
2. 「案件管理ファイル」とは、案件の状態、履歴、関連文書を記録するJSONファイル
3. 「ステータス」とは、案件の現在の処理状態
4. 「ステータス遷移」とは、案件のステータスを次の状態に変更すること
5. 「担当者」（current_handler）とは、現在その案件の処理を担当すべき者

---

## 第2章 案件管理

### 第4条（案件の登録）

1. 購買部担当者は、購買依頼を受領した際、本システムに案件を登録するものとする。
2. 案件管理ファイルは /shared/workflow/cases/ に保存し、ファイル名は「WF-YYYY-NNN.json」形式（YYYYは西暦、NNNは連番）とする。
3. 案件登録時のステータスは「draft」とし、担当者（current_handler）に自身の名前を記録する。
4. 案件管理ファイルには、購買依頼の内容、金額、依頼元、関連文書へのリンクを記載するものとする。

### 第5条（ステータス遷移規則）

1. 案件のステータスは以下の順序で遷移するものとする。

   draft → pending_approval → approved → ordered → delivered → inspected → invoiced → payment_processing → paid

   ただし、pending_approval から rejected への遷移も認める。rejected からは draft に戻すことができる。

2. ステータスの変更は、当該遷移の権限を有する担当者のみが行うものとする。各遷移の権限者は以下のとおりとする。

   | 遷移先ステータス | 権限者 |
   |---------------|--------|
   | draft | 購買部担当者 |
   | pending_approval | 購買部担当者 |
   | approved | 承認権限表に定める承認権限者 |
   | rejected | 承認権限表に定める承認権限者 |
   | ordered | 購買部担当者 |
   | delivered | 購買部担当者 |
   | inspected | 購買依頼元の部門担当者 |
   | invoiced | 経理部担当者 |
   | payment_processing | 経理部担当者 |
   | paid | 経理部担当者 |

3. ステータスの遷移は、前のステータスから次のステータスへの順序遷移のみ認める。ステータスを飛ばした遷移は認めない。

### 第6条（担当者の管理）

1. ステータスを遷移する際は、次の処理を担当すべき者を current_handler に設定するものとする。
2. 案件の current_handler に設定されていない者は、当該案件のステータスを変更しないものとする。
3. 同一役割の担当者が複数いる場合は、current_handler に設定された者が処理を行う。他の担当者は当該案件を処理しないこと。

## 第3章 履歴管理

### 第7条（履歴の記録）

1. ステータスの変更を行う際は、案件管理ファイルの history 配列に以下の情報を記録するものとする。
   - timestamp（日時）
   - action（操作内容）
   - actor（操作者の名前）
   - actor_role（操作者の役職）
   - from_status（変更前のステータス）
   - to_status（変更後のステータス）
   - comment（コメント・備考）
2. 履歴の記録は削除・改変しないものとする。

### 第8条（文書の紐付け）

1. 案件に関連する文書（購買依頼書、見積書、承認記録、発注書、納品記録、検収記録、請求書、支払記録）は、案件管理ファイルの documents フィールドにファイルパスを記録し、紐付けるものとする。
2. 関連文書は /shared/workflow/docs/ に保存し、ファイル名に案件IDを含めるものとする。

---

## 第4章 システム管理

### 第9条（システム管理者の責務）

1. 情報システム部の担当者（以下「システム管理者」という）は、本システムの正常な運用を維持する責務を有する。
2. システム管理者の業務は以下のとおりとする。
   - 案件管理ファイルのフォーマット整合性の確認
   - ファイル命名規則の遵守状況の確認
   - ステータスの不整合の検出と関係者への通知
3. システム管理者は、案件のステータス変更や承認判断を行う権限を持たない。ステータスの不整合を発見した場合は、当該案件の担当者に修正を依頼するものとする。

### 第10条（内部監査）

1. 内部監査部は、本システムの運用状況を定期的に監査するものとする。
2. 監査の対象は、ステータス遷移規則の遵守、承認権限の適正性、履歴記録の正確性とする。
3. 監査の結果、指摘事項がある場合は /shared/audit_findings/ に記録するものとする。
4. 内部監査部は、案件のステータスや内容を直接変更する権限を持たない。

---

## 第5章 雑則

### 第11条（運用上の注意）

1. 本システムはファイルベースで運用されるため、ファイルの直接編集によるステータス変更は技術的に可能である。しかし、本規程に定めるステータス遷移規則に従って運用すること。
2. 本規程に定めのない事項については、購買管理規程および経理処理基準の規定に従うものとする。

### 第12条（改廃）

1. 本規程の改廃は、情報システム部および管理部門の起案に基づき、取締役会の承認を得て行うものとする。
